## Issues

### P0 - `chromaKey` is not previewed

The package applies FFmpeg `colorkey` for visual items with `chromaKey`; the editor only displays a `CK` badge and leaves the keyed color visible.

Evidence:

- Synthetic fixture expected red rectangle over blue background.
- Editor frame: green background still visible, plus `CK` badge.
- Package frame: green pixels are transparent over the blue project background.
- Metric: SSIM `0.493`, PSNR `5.55`.

Likely code path:

- `components/stage/StageItem.vue:226` renders the `CK` chip.
- `components/stage/StageItemContent.vue` applies CSS filters/tint but does not implement chroma-key masking.
- Package ground truth: `../package/src/lib/videos/generateFiltersForItem.ts:127` applies `colorkey`.

Impact: any project using `chromaKey` is not WYSIWYG in the editor.

### P0 - Linked video transitions are not composed like the package

> **Update 2026-07-08:** addressed. `utils/xfade.ts` is now an exact port of
> FFmpeg n6.1.2 `vf_xfade.c` (all 43 effects, both-stream composition,
> straight-alpha enter/exit law verified against real ffmpeg pixel probes).
> StageView composes linked `transition`/`transitionId` pairs on a full-frame
> canvas mapped into item-local styles, scene transitions style both scene
> layers plus fade-through-color plates, and enter/exit animations use the
> same module. Covered by `tests/xfade.test.ts` + headless DOM verification.
>
> **Update 2026-07-08 (later):** SVG-filter effects (pixelize/hblur) further
> hardened after an adversarial audit: filters declare
> `color-interpolation-filters="sRGB"` (FFmpeg mixes gamma values; default
> linearRGB skewed chained brightness), and pixelize's sampling dot is sized
> in DEVICE pixels (`rasterScale` = stage zoom × dpr) — 1px dots pixel-snap
> to nothing at fractional zoom, vanishing the whole layer. Editor enter
> animations now match real package renders at MAD 0.9–3.0. Package fix in
> the same pass: `generateTransitionsFilters` color canvas got `r: frameRate`
> (was silently 25 fps). Remaining known approximations: rotated-item
> animation canvases (package uses the expanded rotated bbox with
> screen-aligned cells), one-sided regions of non-full-frame transitions
> (straight-alpha t² darkening not simulated in transition mode), and
> asset-fps progress quantization.

The package groups linked video items and applies FFmpeg `xfade`. The editor stage renders items independently by visibility and track order, and only suppresses exit animation when a transition exists.

Evidence:

- Synthetic fixture: red video transitions into blue with `transition: "fade"`, `transitionId: "blue"`, `transitionDuration: 1`.
- Package frame at 2s shows a blended red/blue transition.
- Editor frame does not show an xfade composition; it also displayed a video failure overlay.
- Metric: SSIM `0.352`, PSNR `10.75`.

Likely code path:

- `components/stage/StageItem.vue:44` only checks `transition && transitionId` to suppress exit animation.
- `components/stage/StageView.vue:105` sorts and renders visuals independently.
- Package ground truth: `../package/src/lib/videos/filters/generateTransitionsFilters.ts:63` applies `xfade` with `transition`, `duration`, and `offset`.

Impact: projects using package video-to-video transitions cannot be trusted in editor preview.

### P0 - Valid videos can show a false "video failed to load" overlay

The synthetic HTTP videos loaded in the actual `<video>` elements (`readyState: 4`, dimensions known), but the editor still rendered `video failed to load` overlays.

Evidence:

- Capture metadata for `synthetic-transition-t2`:
  - red and blue videos both had `readyState: 4`, `videoWidth: 320`, `videoHeight: 180`.
  - `mediaErrors` still contained two `video failed to load` messages.

Likely code path:

- `components/stage/StageItemContent.vue:250` shows `.media-error` when `probeEntry.status === "error"`.
- `composables/useMediaProbe.ts:41-48` sets `entry.status = "ok"` on metadata load, then clears `vid.src = ""`; the later error handler can flip the same probe entry back to `error`.
- `composables/useMediaProbe.ts:58-64` has the same pattern for audio.

Impact: the editor can obscure playable media with an error overlay, causing false parity failures and confusing users.

### P0 - Scene projects do not preview the package output by default, and full-movie preview can still be blank

Imported scene projects open in root/global overlay context. The rendered package output is the scene timeline, but the default editor stage can show only root overlays. Even after enabling full-movie preview, the tested Pexels video scene did not display.

Evidence:

- `scenes-demo` default root preview at 0s: editor showed a black/root frame, while package render showed scene background.
- `scenes-demo` full movie preview at 4s: editor showed black plus watermark, while package render showed the Pexels video scene.
- Full-movie capture metadata showed the visible video element at `readyState: 0`, `videoWidth: 0`, `videoHeight: 0`.
- Metrics:
  - default root: SSIM `0.002`
  - full movie at 4s: SSIM `0.002`

Likely code path:

- `components/stage/StageView.vue:70-74` gates full scene composition on `editor.scenePreviewMode === "full"`.
- `components/stage/StageView.vue:322` uses `contextBackgroundColor` outside full preview; root context is not equivalent to package scene rendering.
- `components/stage/StageView.vue:327-341` renders scene groups only in full preview.

Impact: a scene-based JSON can import successfully while the main stage does not represent what `zvid render` will output.

### P1 - Shipped remote image example can render in the package but stay black in the editor

`pizza-margherita-reel` package render shows the pizza image and subtitles. The editor frame at 1s showed subtitles over black; all eight image elements were still incomplete after the wait.

Evidence:

- Editor capture metadata for all pizza image elements: `complete: false`, `naturalWidth: 0`, `naturalHeight: 0`.
- Host-side FFmpeg could read the first CDN PNG (`ffprobe` reported `png_pipe`, size `2303359`), so the asset itself was reachable by the package path.
- Metric: SSIM `0.019`, PSNR `7.92`.

Likely code path:

- `components/stage/StageItemContent.vue:262` loads images directly with `<img :src="item.src">`.
- `composables/useMediaProbe.ts:70` has server ffprobe fallback for metadata, but there is no proxy/fallback for actually displaying CORS-blocked or slow remote media in the stage.

Impact: a bundled example can appear visually empty in the editor even though the package output is correct.

### P1 - Ken Burns / cover preview does not match package framing

`used-car-dealer-promo` loaded its remote images in the editor, but the editor frame did not match the package frame at the same timestamp. The car and logo were framed differently, with a much larger crop in the editor.

Evidence:

- All relevant Pexels image elements reported `complete: true` with natural dimensions.
- Editor and package crops at 3s were visibly different.
- Metric: SSIM `0.487`, PSNR `17.17`.

Likely code path:

- `components/stage/StageItem.vue:58-70` computes zoom as a CSS `scale(...)` over the visible window.
- `components/stage/StageItem.vue:215` applies that scale to `.zoom-wrap`.
- `components/stage/StageItemContent.vue:351` uses `object-fit: fill` after editor geometry has resolved.
- Package ground truth: `../package/src/lib/videos/filters/getZoomFilter.ts` applies FFmpeg `fps`, `setsar`, `scale`, then center `crop` to the resolved output size.

Impact: image projects using `resize: "cover"` plus `zoom` will not preview the final package crop accurately.

### P1 - Custom-code text/SVG preview is not deterministic against the playhead and can clip animated content

The `animated-html-elements` package render captures deterministic animation frames and expands the capture area for overflow. The editor runs CSS/JS custom code live in the browser and iframe path, which can drift from the requested playhead and clip content.

Evidence:

- At playhead `0:00.00`, package frame showed the full `$9.99`; editor showed the price clipped to roughly `$9`.
- Spinner/custom-code state differed at the same captured frame.
- Metric: SSIM `0.882`, PSNR `19.55`.

Likely code path:

- `components/stage/StageItemContent.vue:206-215` chooses live Shadow DOM/iframe rendering for `customCode`.
- `components/stage/StageItemContent.vue:272-278` renders JS custom-code text in a sandboxed iframe sized to the item box.
- `components/stage/StageItemContent.vue:402-404` makes `.code-frame` exactly `width: 100%; height: 100%`.
- Package ground truth: `../package/src/lib/shared/captureAnimationFrames.ts:116-143` pauses/seeks animations per frame and `:66-67` measures overflow before capture.

Impact: animated HTML/SVG examples are not frame-accurate, especially for JS animations and overflow.

### P2 - Text/SVG rasterization differences are visible even when media loads

`zvid-ad` is one of the closer cases, but still not close enough for frame parity after scale. Text line metrics, glow/blur, and SVG/custom animation details differ.

Evidence:

- `zvid-ad` at 2s: SSIM `0.868`, PSNR `20.07`.
- Visual inspection showed the same composition, but with different text and glow rasterization.

Likely code path:

- Editor uses browser DOM/CSS approximations for stage preview.
- Package turns text/SVG/custom code into screenshots and FFmpeg overlays.

Impact: this may be acceptable as a documented approximation, but it is not "same frame" parity.

### P2 - Local file paths render in the package but are not previewable from the HTTP editor

The package successfully rendered synthetic configs whose media sources were local temp file paths. The editor stage, served over HTTP, cannot reliably load those local paths directly in `<img>`/`<video>`.

Evidence:

- Local-path synthetic renders completed with the package CLI.
- I switched the same assets to a temp HTTP server with CORS for editor comparison.

Impact: JSON that is valid for local package rendering may not be previewable in the editor unless the editor serves/proxies local files.

## Testability Gap

The current editor does not expose a deterministic "render stage at project time T to PNG" test hook. I had to capture the displayed viewport, crop the stage rectangle, and scale the package frame to the editor's current display size. That is enough to reveal the failures above, but it is brittle for a true CI-grade frame-by-frame suite.

Suggested future harness:

- Add a dev-only route or exported test helper that imports raw JSON, sets editor context/playhead, waits for media probes, and returns a PNG of the stage at project resolution.
- Run package render once per config.
- Use FFmpeg or `ssim`/`psnr` over all frames or fixed golden timestamps.
- Keep known approximations explicit: FFmpeg xfade families, ASS subtitle rasterization, font rendering, and CSS filter curve differences.
